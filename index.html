Decentralized storage of annotations

Intermediate thesis presentation by Lukas Vanhoucke

Current social media networks

  • All data is stored in a centralized manner
    • EULA usually states that you do not own the stored data
    • Need to create new account and data for each social network (information silos)
    • Granularity of access rights to data is dependent on social network
  • Annotation systems are a kind of social network, since it involves users interacting with eachother.
  • For practical reasons, annotation systems are almost always centralized.
  • Is it possible to decentralize annotation systems to solve the issues stated above?

Decentralization of annotation systems

  • By using Linked Data principles, decentralization can be achieved.
    • Solid for storage and CRUD operations
    • Linked Data Notifications
  • Biggest source of inspiration: dokieli

Research questions

  • “Is it possible to design a workflow to allow retrieval and storage of annotations, and sending/retrieving notifications about updates in a fully decentralized manner, using Linked Data principles?”

      Most challenging: notifying the inbox of a server, and how the server should process this.

  • “Can this workflow be simplified by providing a JavaScript library that generically implements decentralized annotation functionality, and can easily be added to existing annotation systems?”

Research questions

  • Possibly: “Would this library benefit from a Linked Data Notifications library that implements the LDN standards? If this does not exist, is it worth it to write it yourself?”
  • How do I measure this?

Hypothesis

  • “Can decentralized storage of annotations be implemented in an equally fast and easy to use manner compared to centralized annotations?”
  • Again, how to quantify?
    • “Fast”: The user should not notice a difference
    • "Easy to use":
      • Creating an account on a website <-> Creating a WebID personal storage
      • Commenting anonymously <-> Writing to an open inbox
